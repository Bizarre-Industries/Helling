# Helling v0.1 Roadmap

> Status: in progress. Last updated 2026-05-02.
> Owner: @binGhzal

## Goal

Ship a runnable, end-to-end vertical slice that proves the architecture: a single user can log in via the web UI, create a Debian container, see it running, stop it, delete it. That's the bar.

If a feature isn't on this list, it's post-v0.1.

## Definition of done

A v0.1 build is releasable when **all** of the following are true:

- [ ] Single-host install on Debian 13 works via `dpkg -i` or `tar.gz` + systemd unit
- [ ] `make check` passes locally and in CI
- [ ] `make check-generated` passes (no spec drift)
- [ ] All endpoints in `api/openapi.yaml` are implemented and tested
- [ ] Web dashboard ships with: login page, instance list, instance detail, instance create form
- [ ] CLI ships with: `helling login`, `helling logout`, `helling instance ls/create/start/stop/rm`
- [ ] At least one non-trivial integration test runs against a real Incus instance
- [ ] Vulnerability scan (govulncheck + gosec) is clean
- [ ] `docs/spec/architecture.md` matches the implementation
- [ ] An external person can install and use it following only `README.md` + `docs/install.md` (the latter doesn't exist yet — write it)

## Out of scope (deferred to v0.2 or later)

- VM (KVM) instances. Containers only in v0.1.
- Storage volume management
- Network customization beyond the Incus default bridge
- Multiple users with isolation between them
- OIDC / SSO
- API tokens for non-interactive auth (CLI uses session cookie via `~/.config/helling/session`)
- Image upload (use Incus's image server / aliases only)
- Snapshots
- Live migration
- Metrics / Prometheus
- Distributed tracing
- Multi-host clustering
- Web UI dark mode (default antd theme is fine for now)

## Known risks

1. **Incus's async operation model is hostile to typed clients.** `Operation` is a runtime object with method receivers, not a JSON shape. We re-shape into our own type at the boundary. Risk: mistakes there leak Incus internals into our API. Mitigation: integration test that asserts no Incus types appear in any OpenAPI-shaped response.
2. **CGO temptation.** PAM and `mattn/go-sqlite3` both pull CGO in. We rejected PAM; we're defaulting to pure-Go SQLite. If perf forces a switch, accept the cross-compile and Docker complexity.
3. **OpenAPI 3.0 is feeling its age.** `oneOf`/`anyOf` discriminators are awkward. Workarounds: keep v0.1 schemas dumb, defer composition to v0.2 when oapi-codegen's 3.1 support catches up.
4. **Rate-limit state lost on restart.** v0.1 keeps it in-memory. Acceptable for single-node; revisit when we add HA.
5. **No password reset flow** without admin involvement. Will produce real support load. Plan email-based reset for v0.2.

## What success looks like

A user clones the repo, runs `make dev-setup && make build`, runs `./bin/hellingd`, runs `./bin/helling-proxy`, opens `https://localhost`, logs in, creates a Debian container, and sees it running, all in under 10 minutes on a fresh Debian 13 VM. If that flow doesn't work, v0.1 isn't done.
